# AI 에이전트 발전 과정

LLM 기반 크레인 게임 에이전트를 개발하면서 겪은 시행착오와 인사이트를 기록한다.

---

## 1. 초기 시도 — 단순 프롬프트

처음에는 LLM에게 스크린샷과 함께 "빨간 공을 집어라"라는 단순한 명령만 전달했다.

**발생한 문제:**
- 사선(isometric) 카메라 시점에서 깊이를 정확히 판단하지 못함
- 공이 크레인 뒤에 가려져 있을 때 위치를 파악하지 못함
- 카메라 한 각도로는 2차원 평면의 공 위치를 정확히 알 수 없음

**교훈:** LLM의 시각 이해 능력은 뛰어나지만, 단일 시점의 한계는 사람과 동일하다.

---

## 2. 카메라 도구 추가

에이전트에게 카메라 회전 도구를 부여했다. 90도 단위로 회전하여 다른 각도에서 장면을 관찰할 수 있게 되었다.

**효과:**
- 한 각도에서 가려진 공도 다른 각도에서 확인 가능
- 깊이 정보를 좌우 정보로 변환하여 관찰 가능

**교훈:** 행동 도구(move, grip)만큼 **관찰 도구(camera)**도 중요하다. 정보 부족은 잘못된 행동의 근본 원인이 된다.

---

## 3. 전략 프롬프트 도입

단순히 "무엇을 하라"(공을 잡아라)가 아니라, **"어떻게 하라"**(구체적인 전략)를 프롬프트에 포함시켰다.

### 2축 정렬 전략

사선 카메라에서는 forward/backward(깊이 방향) 판단이 부정확하다. 그래서 다음 전략을 프롬프트에 명시했다:

1. 현재 화면에서 **좌우(left/right)만 정렬**
2. 카메라를 **90도 회전** — 이전의 깊이 차이가 좌우로 변환됨
3. 새로운 화면에서 **좌우만 정렬**
4. 두 축이 모두 정렬되면 → 내리기 → 집기 → 올리기

**forward/backward 사용을 금지**하고, 카메라 회전을 통해 모든 이동을 좌우로만 처리하도록 강제했다.

**효과:** 축 혼동이 크게 줄고, 정렬 정확도가 향상되었다.

**교훈:** LLM 에이전트에게는 "목표"보다 "방법론"을 알려주는 것이 더 효과적이다. 사람에게 매뉴얼을 주는 것과 같다.

---

## 4. 메모(Memo) 도구 추가

에이전트가 자신의 관찰 결과, 계획, 진행 상황을 텍스트로 기록할 수 있는 `memo` 도구를 추가했다. 메모는 매 스텝 컨텍스트에 포함되어 에이전트가 참조할 수 있다.

```
예시 메모: "1차 좌우 정렬 완료. 카메라 회전하여 전후 축 확인 필요."
```

**메모가 없을 때의 문제:**
- 대화 히스토리만으로는 "지금 어떤 단계인지" 파악이 어려움
- 카메라 회전 후 이전에 맞춘 축을 다시 건드리는 실수
- 같은 행동을 반복하는 루프

**메모 도입 후:**
- 상태 추적이 명확해져 일관된 행동 가능
- "1차 정렬 완료" → "카메라 회전" → "2차 정렬" 순서를 기억
- 자기 참조(self-reference)를 통한 계획 실행력 향상

**교훈:** 컨텍스트 윈도우에는 한계가 있다. 에이전트가 스스로 구조화된 메모를 남기면, 장기 계획 실행력이 크게 향상된다. 이는 사람이 작업 중 메모를 남기는 것과 같은 원리다.

---

## 5. 도구 호출 강제 — Retry 메커니즘

LLM이 도구를 호출하지 않고 텍스트만 출력하는 경우가 간헐적으로 발생했다. 예를 들어 "공이 왼쪽에 있으므로 왼쪽으로 이동해야 합니다"라고 말만 하고 실제 move 도구를 호출하지 않는 케이스.

**해결:**
- 첫 응답에서 tool_calls가 없으면, "반드시 도구를 호출해야 합니다"라는 안내와 함께 자동 재시도
- 재시도 메시지는 히스토리에 남기지 않음 (오염 방지)

**교훈:** LLM 에이전트 시스템에서는 "LLM이 항상 올바르게 동작한다"고 가정하면 안 된다. 안전장치와 폴백이 필요하다.

---

## 6. 디버그 로깅 — 전체 메시지 히스토리

에이전트가 "공을 잡았다"고 `done`을 호출했지만, 실제로는 공을 잡지 못한 경우가 있었다. 원인을 진단하려면 에이전트가 무엇을 보고, 어떻게 판단했는지 전체 맥락이 필요했다.

**구현:**
- 매 스텝의 전체 메시지 히스토리(시스템 프롬프트, 스크린샷 여부, LLM 응답, 도구 호출, 도구 결과)를 저장
- 대시보드에서 스텝별로 전체 메시지를 열람 가능
- JSON 형태로도 저장하여 향후 지식 증류(Knowledge Distillation) 학습 데이터로 활용 가능

**교훈:** 에이전트의 판단 과정을 투명하게 기록하는 것은 디버깅뿐 아니라, 더 작고 효율적인 모델을 훈련하기 위한 데이터 파이프라인의 기반이 된다.

---

## 7. 이미지 토큰 절약

매 스텝마다 스크린샷이 메시지 히스토리에 추가되면서 토큰이 기하급수적으로 증가했다.

**해결:**
- 최근 2개의 스크린샷만 원본 유지
- 이전 스크린샷은 `[이전 스크린샷]` 텍스트로 대체
- 텍스트 컨텍스트(프롬프트, 도구 결과, 메모)는 전부 유지

**교훈:** 멀티모달 에이전트에서 이미지는 가장 큰 토큰 소비원이다. "최근 N개만 유지" 전략은 성능 저하 없이 비용을 절감하는 효과적인 방법이다.

---

## 핵심 인사이트 요약

| 인사이트 | 설명 |
|---------|------|
| **전략 > 목표** | "공을 잡아라"보다 "이렇게 잡아라"가 효과적. 방법론을 알려주면 성능이 오른다. |
| **관찰 = 행동** | 카메라 같은 관찰 도구는 행동 도구만큼 중요. 정보 부족이 오판의 원인. |
| **자기 참조** | 메모 도구로 에이전트가 자신의 상태를 추적하면 장기 계획 실행력이 향상된다. |
| **안전장치 필수** | LLM은 가끔 도구를 호출하지 않는다. retry, fallback 등 방어 코드가 필요하다. |
| **투명한 로깅** | 전체 메시지 히스토리를 저장하면 디버깅과 지식 증류 모두에 활용 가능하다. |
| **토큰 관리** | 이미지 누적은 비용 폭발의 원인. 최근 N개만 유지하는 전략이 효과적이다. |
